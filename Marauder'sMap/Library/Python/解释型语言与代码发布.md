# 解释型语言与代码发布

## TL;DR

Python 和 JavaScript 都是**解释型语言**，发布程序 = 发布源代码。前端 JS 更没隐私（浏览器里 F12 直接看）。想保护代码，要么打包二进制（仅限后端），要么用 WebAssembly，要么做成服务端 API。

---

## 1. 编译型 vs 解释型

```text
编译型（C/Java）:  源代码 → 编译 → 二进制文件 → 运行
                   发布的是二进制，别人看不到源代码

解释型（Python/JS）: 源代码 → 解释器逐行执行
                     发布的就是源代码本身
```

类比：
- 编译型 = 把菜谱做成一道菜，别人吃到菜但拿不到菜谱
- 解释型 = 直接把菜谱发给别人，别人既能做菜也能看到配方

---

## 2. Python 发布 = 发布源代码

Python 的 `.py` 文件就是源代码，打开就能看到每一行代码。

`.pyc` 文件（字节码缓存）只是加速加载用的，很容易被反编译回接近原始的源代码，起不到保护作用。

详见 [[CPython 解释器原理]]。

---

## 3. JavaScript 也是一样

| 场景 | 情况 |
|------|------|
| 前端 JS | **完全公开**，浏览器 F12 就能看到 |
| 后端 JS (Node.js) | 和 Python 一样，部署的就是 `.js` 文件 |

前端 JS 几乎必做混淆/压缩（Webpack/Terser），但只是把变量名变成 `a, b, c`、去掉空格换行，本质还是源代码，有耐心就能还原。

---

## 4. 保护手段对比

| 手段 | Python | JavaScript |
|------|--------|------------|
| 混淆/压缩 | 不常见 | 前端几乎必做 |
| 打包成二进制 | PyInstaller, Nuitka | pkg, nexe |
| 做成服务端 API | 常见 | 常见 |
| WebAssembly | 可以（通过 C 扩展） | 可以 |

打包成二进制只能用于**后端/桌面端**，浏览器不能运行二进制文件。

---

## 5. WebAssembly (WASM)

浏览器只认三样东西：**HTML、CSS、JavaScript**。WASM 是第四种——一种浏览器能运行的二进制格式：

- 用 C/C++/Rust 写代码 → 编译成 `.wasm` → 浏览器执行
- 比 JS 快，且很难反编译回可读源代码
- 适合需要性能或保护逻辑的场景（游戏引擎、加密算法）

### WASM 不能替代 JS

WASM **不能直接操作网页（DOM）**，必须通过 JS 当"中间人"：

| 能力 | JS | WASM |
|------|-----|------|
| 修改页面元素 | 直接操作 | 必须调 JS |
| 监听点击/输入事件 | 直接监听 | 必须通过 JS |
| 发网络请求 | `fetch()` | 必须通过 JS |
| 高性能计算 | 较慢 | 很快 |

类比：
- JS = 自己去餐厅点菜，能直接跟服务员（DOM）说话
- WASM = 外国人，必须带个翻译（JS）才能跟服务员沟通

现实中的分工：**JS 负责页面交互，WASM 负责重计算**（图像处理、视频编解码、3D 渲染等），两者配合。

---

## 6. 最靠谱的保护方式

不发给用户——放在服务端，只暴露 API。这也是 SaaS 模式流行的原因之一。
